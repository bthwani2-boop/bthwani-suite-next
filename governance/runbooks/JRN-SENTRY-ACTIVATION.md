# Mobile Sentry activation runbook

## Scope

This runbook governs Sentry activation for `app-client`, `app-partner`, `app-captain`, and `app-field`.
Sentry is a technical observability provider only. It is not an identity, tenant, finance, analytics, or audit source of truth.

## Required EAS variables per application and environment

- `EXPO_PUBLIC_SENTRY_DSN`: project DSN exposed to the application.
- `SENTRY_AUTH_TOKEN`: secret token used only by the build/update job to upload source maps.
- `SENTRY_ORG`: Sentry organization slug.
- `SENTRY_PROJECT`: application-specific Sentry project slug.
- `EXPO_PUBLIC_APP_ENV`: `development`, `preview`, or `production`.
- `EXPO_PUBLIC_SENTRY_TRACES_SAMPLE_RATE`: bounded number from `0` through `1`; production starts conservatively.

Each mobile application uses its own Sentry project. Credentials must be stored in the matching EAS project and environment; they must never be committed to Git.

## SaaS privacy boundary

Sentry events must not contain:

- access, refresh, activation, service, or provider tokens;
- cookies or authorization headers;
- raw `X-Tenant-ID` or client-asserted ownership context;
- phone numbers, email addresses, bank data, payout destinations, identity documents, precise location history, or message bodies;
- wallet balances, ledger payloads, payment credentials, or unmasked financial references.

Stable opaque correlation IDs and application surface names are allowed. Tenant-aware diagnosis must use a non-reversible operational alias generated by trusted backend telemetry, not a client-supplied tenant identifier.

## Release verification

Before production activation:

1. Run the mobile EAS preflight from an immutable clean commit.
2. Confirm the Sentry Expo config plugin is present in the resolved Expo config.
3. Confirm Metro is generated through `getSentryExpoConfig`.
4. Create one internal build and capture a controlled test exception.
5. Verify the issue contains symbolicated JavaScript/native frames and the expected environment/release.
6. Verify redaction by inspecting the event payload for forbidden fields.
7. Verify EAS Update source maps against the `preview` channel before enabling production updates.

## Failure policy

Missing DSN disables event delivery without crashing the application. Missing build-time source-map credentials blocks preview/production preflight. Production must not proceed with unsymbolicated crash reporting.
