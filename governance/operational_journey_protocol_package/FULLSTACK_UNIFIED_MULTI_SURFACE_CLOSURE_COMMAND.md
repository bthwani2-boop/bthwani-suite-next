نفّذ تشخيصًا جنائيًا عميقًا وفق أفضل الممارسات والمعايير الرسمية والمصادر المفتوحة الموثوقة لكل ما يرتبط بالتنفيذ فعليًا، بما يشمل الكود الحي والملفات والمسارات والطبقات والخدمات والتطبيقات والأسطح وأقسام لوحة التحكم والشاشات والصفحات والتبويبات والمسارات والأزرار والأيقونات والصور والعناصر التفاعلية والوظائف والبيانات والعقود والعملاء المولدين والتبعيات والمنطق والسلوك والتشغيل وقواعد البيانات والصلاحيات والاختبارات والحراس وبيئة Runtime، دون الاعتماد على المحلي أو الذاكرة أو الانطباع أو أي افتراض غير مثبت.

يجب أن يعمل المشروع كنظام **Full-Stack موحد متعدد الأسطح**، لا كمجموعة تطبيقات منفصلة. ويجب ضمان التطابق والربط الرأسي الكامل بين الفرونت إند والباك إند والعقود والعملاء المولدين وقاعدة البيانات والحقيقة التشغيلية والمالية وجميع الأسطح؛ وأي انفصال أو تعارض أو نقص أو اختلاف في الطلبات أو الاستجابات أو الحقول أو الحالات أو الصلاحيات أو الأخطاء أو الآثار التشغيلية أو القراءة الراجعة يُعد خللًا حرجًا يجب تصحيحه، وتبقى النتيجة `FIX_REQUIRED`.

يجب أن يكون كل زر أو تبويب أو أيقونة أو صورة تفاعلية أو رابط أو شاشة أو صفحة أو نموذج أو حقل أو ميزة أو route أو endpoint أو وظيفة مربوطًا فعليًا من الواجهة إلى المصدر السيادي الحقيقي ثم عائدًا بالنتيجة الصحيحة إلى جميع الأسطح المتأثرة. لا يجوز أن يعمل أي سطح بصورة منفصلة، أو أن يمتلك حقيقة أو منطق أعمال أو حالات أو صلاحيات أو أنواع بيانات مستقلة عن بقية النظام.

أثناء التشخيص والتنفيذ، افترض دائمًا أن المطلوب قد يكون موجودًا جزئيًا أو كليًا في ملفات أخرى. لذلك، قبل إنشاء أي ملف أو مكوّن أو hook أو controller أو endpoint أو schema أو type أو helper أو منطق جديد، افحص الموجود فعليًا عبر البحث الشامل والتبعيات والاستيرادات والتصديرات والمسارات والاستخدامات والمستهلكين والتشابه الوظيفي والعقود والعملاء المولدين والـregistries والـdependency graph، ثم فضّل إعادة الاستخدام أو التوسيع أو الدمج أو النقل إلى المالك الصحيح بدل إنشاء بدائل مكررة. لا تنشئ ملفًا جديدًا إلا بعد إثبات عدم وجود ملف حالي مناسب، أو إثبات أن تعديل الموجود سيزيد الخطر أو التشتت، مع توضيح سبب الإنشاء ومالكه وموقعه ومستهلكيه وطريقة التحقق منه.

عالج جميع مصادر التشتت وتعدد الحقائق والمصادر والتناقض والتكرار والتسرب بين الطبقات والخدمات والأسطح. استخرج المشكلات من جذورها وحدد بدقة:

- ما الصحيح.
- ما الخاطئ.
- ما الناقص.
- ما غير المربوط.
- ما يعمل شكليًا فقط.
- ما يعمل في سطح ويتعطل أو يتناقض في سطح آخر.
- ما المكرر.
- ما الميت.
- ما المتسرب.
- ما الموجود في غير مالكه.
- ما لا يطابق العقود أو قاعدة البيانات.
- ما يعتمد على mock أو demo أو preview بدل الحقيقة التشغيلية.
- ما لا يمتلك اختبارًا أو دليل Runtime كافيًا.
- ما يجب تصحيحه أو إضافته أو تحديثه أو نقله أو دمجه أو تنظيفه فورًا.

لكل مشكلة، حدّد مكانها وسببها الجذري وأثرها والأسطح المتأثرة بها وأولويتها ومالكها الصحيح وإجراء معالجتها وطريقة التحقق منها. حوّل أي غموض إلى نقص محدد يوضح ما الدليل المفقود، ولماذا يمنع الحسم، وكيف يمكن إثباته.

لا تحذف أو تنقل أو تدمج أي ملف أو مكوّن أو route أو export أو وظيفة إلا بعد إثبات عدم ارتباطه بالكود الحي أو التشغيل أو العقود أو الواجهات أو الاختبارات أو الحراس أو CI أو التنقل أو الـregistries أو المستهلكين أو الأحداث أو الوظائف المجدولة. لا يُحذف إلا ما ثبت بدليل أنه مكرر أو ميت أو ضجيج غير مستخدم ولا يضيف قيمة فعلية.

ركّز على معالجة وتصحيح الكود الحي وربط جميع الشاشات والتبويبات والأزرار والأيقونات والوظائف والرحلات غير المربوطة بآلية عملية ومنظمة وواضحة من الألف إلى الياء، مع تنظيف الكود بحذر من التكرار والضجيج والمخلفات، دون تحويل التنفيذ إلى تقارير أو مصفوفات أو ملفات أدلة كثيرة بدل الإصلاح الفعلي.

يُشترط تشغيل وفحص كافة مسارات سير العمل (Workflows) المعرّفة في المجلد [.github/workflows](file:///c:/bthwani-suite-next/.github/workflows) التي تقع ضمن سياق التنفيذ الحالي. وبناءً على مخرجات هذه الـ Workflows وتقارير الفشل المرتبطة بها، يتعين إجراء تشخيص جنائي عميق وجذري لتحديد مكامن الخلل، ومن ثم تصحيح ومعالجة ملفات ومجلدات الكود الفعلي مباشرة في المستودع لضمان استقرار البناء والتكامل قبل إعلان الجاهزية.

اختر المعالجة الأقل مخاطرة والأعلى دقة، ثم نفّذ الإغلاق الفعلي لجميع الفجوات القابلة للإصلاح داخل المستودع. أعد الجرد والفحص والربط والتحقق بعد كل معالجة، ولا تعتبر نجاح `build` أو `typecheck` أو اختبار فرونت إند أو باك إند منفردًا دليلًا على اكتمال الفول ستاك؛ بل يجب إثبات الرحلة كاملة من عنصر الواجهة إلى العقد والباك إند وقاعدة البيانات ثم القراءة الراجعة الصحيحة في جميع الأسطح المتأثرة.

بعد استكمال التشخيص والفحص والتحليل العميق والدقيق والتقني والمنطقي والتشغيلي وكل شيء، يجب كتابة أمر التنفيذ بدقة رقمية كاملة 100%، مع التزام صارم بـ صفر أخطاء، صفر تناقض، صفر تكرار، صفر فجوات، صفر ضجيج، وصفر ضعف أو نقص في المنطق والتشغيل، صفر تشتت، صفر فشل، وصفر عيوب. بحيث نضمن الأدق والأقوى والأفضل والأنسب دون أي ضعف أو نقص، وبمنهجية حقيقية، منطقية، وتقنية محقَّقة بالكامل 100%.

لا تعلن الجاهزية أو الإغلاق أو `IMPLEMENTATION_PASS` أو `100%` إلا بعد تحقق عملي يثبت:

- صفر أخطاء.
- صفر تناقض.
- صفر تكرار غير مبرر.
- صفر فجوات داخل النطاق.
- صفر كود ميت داخل النطاق.
- صفر ضجيج غير مفيد.
- صفر تسرب بين الطبقات أو الخدمات.
- صفر عناصر واجهة غير مربوطة.
- صفر ميزات تعمل في سطح بصورة منفصلة.
- صفر اختلاف بين الفرونت إند والباك إند.
- صفر اختلاف بين العقود والتنفيذ والعملاء المولدين.
- صفر اختلاف بين العقود وقاعدة البيانات.
- صفر اختلاف في الحالات أو الصلاحيات أو الأخطاء.
- صفر نجاح شكلي في الواجهة دون أثر خلفي حقيقي.
- صفر أثر خلفي دون قراءة راجعة وتحديث للأسطح المتأثرة.
- صفر ضعف منطقي أو تشغيلي قابل للإصلاح داخل النطاق.

أي فجوة أو عدم تطابق أو انفصال أو عنصر غير مثبت يبقي النتيجة `FIX_REQUIRED`. وأي أمر تعذر إثباته بسبب وصول أو سر أو جهاز أو خدمة أو بيئة خارجية يجب تصنيفه صراحةً `BLOCKED_NEEDS_EVIDENCE`، ولا يجوز تحويله إلى نجاح أو تجاهله.
