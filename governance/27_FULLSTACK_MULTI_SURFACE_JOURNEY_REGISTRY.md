# 27 — سجل الرحلات الموحدة متعددة الأسطح

Status: ACTIVE_CANONICAL

Inventory mode: LIVING

Registry ID: `BTHWANI-FULLSTACK-JOURNEYS`

Initial baseline branch: `sambassam`

Initial baseline commit: `466d34bacdd7fde4cc1c351e094d3d4604b33268`

## الغرض

هذا الملف هو السجل الحي الموحد لأسماء الرحلات التشغيلية وشرائحها في `bthwani-suite-next`. يجب الرجوع إليه عند التخطيط والتنفيذ والمراجعة، وتحديثه كلما ظهرت قدرة أو عقد أو مسار أو شاشة أو انتقال حالة جديد.

هذا السجل لا يثبت أن رحلة ما مكتملة، ولا يتجاوز Product Truth أو العقود أو manifests أو الكود أو قاعدة البيانات أو الأدلة الفعلية. عند التعارض تكون الحقيقة للمرجع الأعلى حسب `governance/authority/authority-precedence.json` وللحالة التنفيذية في الـcommit المثبت.

## النطاق

- النوى المشتركة: `identity`, `workforce`, `platform-control`, `providers`.
- الخدمات الفعلية الحالية: `dsh`, `wlt`.
- الأسطح المسجلة: `app-client`, `app-partner`, `app-captain`, `app-field`, `control-panel`, `webapp`, `website`.
- لا يلزم ظهور كل رحلة في كل سطح؛ يجب تحديد الأسطح المطلوبة والمستبعدة مع السبب في Product Truth الخاص بالرحلة.
- مجالات الخدمات `knz`, `arb`, `amn`, `esf`, `mrf`, `snd`, `kwd` لا تُعد رحلات فعالة حتى يثبت تفعيلها بmanifest وعقد ومسار وتنفيذ. عند تفعيل أي منها يجب إضافة رحلاتها هنا قبل الدمج.

## طريقة استخدام السجل وتحديثه

1. ثبّت المستودع والفرع و`resolved_commit_sha` قبل أي تحليل أو كتابة.
2. اختر `Journey ID` واحدًا وعرّف نطاقه والأسطح المطلوبة والمستبعدة قبل التنفيذ.
3. حدّث الشرائح الوظيفية هنا إذا كشف التشخيص عقدًا أو مسارًا أو حالة أو شاشة غير مسجلة.
4. نفّذ شرائح قصيرة متسلسلة، ولا تنتقل للرحلة التالية عند وجود قرار إيقاف.
5. بعد كل شريحة سجّل commit الدليل والقرار والفجوات المتبقية في قسم سجل التنفيذ.
6. لا تحذف رحلة تاريخية؛ استخدم `MERGED_INTO`, `RETIRED` أو `OUT_OF_SCOPE_FOR_THIS_JOURNEY` مع المرجع والسبب.
7. أي عقد أو operation أو migration أو route أو شاشة جديدة غير ممثلة هنا تُعد فجوة سجل ويجب إضافتها قبل الدمج.
8. لا تُستخدم `CLOSED_WITH_EVIDENCE` إلا بعد نجاح كل نطاقات الأدلة المنطبقة على نفس commit وبالموافقات المستقلة المطلوبة.

## حالات التتبع والقرارات

حالة التتبع الداخلية: `NOT_ASSESSED`, `IN_PROGRESS`, `READY_FOR_REVIEW`, `PAUSED`, `MERGED_INTO`, `RETIRED`.

القرار الحاكم يجب أن يكون أحد قرارات `governance/contracts/decision-vocabulary.json`. جميع الرحلات أدناه تبدأ بـ`NEEDS_EVIDENCE` حتى تُقيّم على commit ثابت؛ هذا لا يعني أن التنفيذ غير موجود، بل يعني أن هذا السجل وحده ليس دليلًا.

## تفاصيل الرحلات والشرائح الوظيفية

### JRN-001 — تأهيل الشريك والمتجر ونشره

الأسطح المرشحة: `app-field`, `app-partner`, `control-panel`.

- إنشاء مسودة الشريك ميدانيًا وربط الهوية القانونية وبيانات الاتصال.
- ترشيح المتجر وربط المدينة والمنطقة والموقع وساعات العمل.
- الوثائق والصور والفيديو والأدلة ورفع الوسائط وقراءتها.
- الزيارات الميدانية وفحوص الجاهزية وتسجيل النواقص.
- مراجعة الوثائق وقبولها أو رفضها مع السبب.
- انتقالات حالة الشريك والمتجر والجاهزية للنشر.
- ربط الحسابات والمتاجر والنطاقات والصلاحيات.
- دعوات فريق المتجر وقبولها ورفضها وإدارة الأعضاء.
- التفعيل والنشر والإخفاء والتعليق وإعادة التفعيل.
- سجل التدقيق وقراءة الحالة الذاتية للشريك.

### JRN-002 — الهوية والتفعيل والجلسات

الأسطح المرشحة: جميع الأسطح ذات تسجيل الدخول، و`control-panel` للإدارة.

- Health وReadiness لخدمة Identity.
- Provision للـActor والبحث الداخلي وجلب التفاصيل.
- طلب OTP أو إصدار رمز تفعيل لممثل محدد.
- عرض أحدث تفعيل وسحب جميع رموز التفعيل.
- استهلاك رمز التفعيل وتسجيل الدخول.
- إصدار وتدوير Refresh Token واسترجاع الجلسة الحالية.
- introspection وعرض الجلسات وإلغاء جلسة بعينها.
- تسجيل الخروج وتغيير كلمة المرور.
- تعطيل Actor وإعادة تفعيله.
- حذف الحساب وإخفاء الهوية وإدارة outbox المرتبط.
- فرض الدور والصلاحية والسطح ومنع actor leakage.
